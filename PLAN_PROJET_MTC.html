Architecture technique du projet

Technologies utilisées

ComposantTechnologieRôle
Application desktopElectron v28Fenêtre native Windows / Mac / Linux
Interface utilisateurReact 18 + TypeScriptToutes les pages et composants visuels
Base de donnéesSQLite via better-sqlite3Stockage local des patients, séances, RDV
ChiffrementAES-256-GCM (Node.js crypto)BDD au repos + sauvegardes
ConstructionVite + electron-builderCompilation et packaging installateurs
StylesCSS vanilla (global.css)Design de l'interface
💡 Architecture 2 processus Electron : Le Main Process (Node.js) gère la base de données, les fichiers, le chiffrement et les IPC. Le Renderer Process (React) gère l'interface. Ils communiquent via des canaux IPC sécurisés exposés par le preload.

🔐 Sécurité & Protection des données

🔒 Mot de passe + Chiffrement AES-256-GCM de la base de données
  • 🔑Écran de connexion par mot de passe au démarrage — aucun accès aux données sans le mot de passe
  • 🔐Base de données SQLite chiffrée en AES-256-GCM avec clé dérivée via PBKDF2 (600 000 itérations) — le fichier mtc.sqlite est illisible sans mot de passe
  • ⏱️Verrou automatique après 20 minutes d'inactivité — l'écran de connexion réapparaît
  • 💾Re-chiffrement automatique toutes les 3 minutes pendant la session (auto-save sécurisé)
  • 🚪À la fermeture : la base est chiffrée, le fichier de travail déchiffré est supprimé
  • 🔄Changement de mot de passe depuis Paramètres → re-chiffrement immédiat avec le nouveau mot de passe
  • 🛡️Badge 🔒 chiffré visible dans le header en permanence

📅 Calendrier & Rendez-vous

📅 Calendrier complet avec synchronisation bidirectionnelle
  • 🗓️Calendrier mensuel avec vue planning horaire par jour (créneaux de 30 min, de 08h00 à 19h30)
  • ⏰Sélection de l'heure au créneau de 5 minutes dans les sélecteurs (de 07h00 à 20h00)
  • 🎨Code couleur des RDV : bleu = planifié, vert = réalisé, orange = en attente
  • 👤Prise de RDV pour nouveau patient sans fiche : saisie nom, prénom, téléphone, motif de venue directement dans le calendrier
  • 👥Bouton Créer la fiche patient depuis un RDV invité : crée automatiquement la fiche et lie le RDV
  • 🔁Synchronisation bidirectionnelle : le prochain RDV saisi dans Nouvelle séance crée/met à jour un RDV dans le calendrier, et inversement
  • 📋Heure début + heure fin + motif + note dans le formulaire de RDV
  • 📋Bouton Créer la séance depuis un RDV : ouvre Nouvelle séance avec le motif et la date pré-remplis

Feuille de route — Évolution du projet

Projet d'évolution vers un logiciel multi-thérapeutes, modulaire et commercialisable par spécialité.

1
Système de plugins par spécialité
⏱️ Estimation : 5 à 7 semaines · Priorité : HAUTE
Architecture modulaire où l'anamnèse est définie dans un fichier .plugin (JSON) séparé du logiciel core. Chaque spécialité (MTC, Kinésiologie, Ostéopathie, Naturopathie) a son propre fichier plugin. Le moteur de formulaire dynamique dans le core lit le plugin et génère le formulaire automatiquement.

Ce que ça permet : vendre le logiciel par spécialité — un installateur MTC, un installateur Kinésiologie, etc. Mettre à jour une anamnèse sans livrer une nouvelle version du logiciel core.
2
Plugin Kinésiologie — Anamnèse personnalisée
⏱️ Estimation : 3 à 5 jours · Priorité : HAUTE
Création du fichier kinesio.plugin à partir de l'anamnèse fournie par la kinésiologiste. Premier plugin non-MTC de l'application. Valide le moteur de formulaire dynamique en conditions réelles.

Prérequis : recevoir l'anamnèse kinesio complète (format papier, Word ou PDF).
3
Packaging multi-spécialités
⏱️ Estimation : 2 à 3 jours · Priorité : HAUTE
Script de build qui génère automatiquement un installateur dédié par spécialité : Setup-MTC-v1.1.0.exe, Setup-Kinesio-v1.1.0.exe, etc. Chaque installateur contient le core + le plugin de la spécialité, et rien d'autre. Le thérapeute installe un seul exe et voit directement son formulaire.
4
Améliorations RGPD
⏱️ Estimation : 1 à 2 semaines · Priorité : MOYENNE
Compléter la conformité RGPD pour les données de santé :
• Case à cocher consentement patient + date sur chaque fiche
• Notice d'information patient (texte configurable, impression possible)
• Journal d'accès minimal (date/heure d'ouverture des fiches)
• Durée de conservation configurable avec alertes
• Registre des traitements exportable (Art. 30 RGPD)
5
Multi-utilisateurs (cabinet partagé)
⏱️ Estimation : 4 à 6 semaines · Priorité : SELON BESOIN
Permettre à plusieurs thérapeutes d'un même cabinet d'accéder à une base partagée sur réseau local. Nécessite de remplacer SQLite par PostgreSQL sur un PC serveur du cabinet, d'ajouter la gestion de comptes utilisateurs avec rôles (praticien / secrétaire) et de gérer les accès simultanés.

Note : si chaque thérapeute travaille de façon indépendante (cabinet personnel), cette fonctionnalité n'est pas nécessaire — chacun installe sa propre version.
6
Version tablette / mobile (Capacitor)
⏱️ Estimation : 4 à 6 semaines · Priorité : SELON BESOIN
Portage vers une application iOS / Android avec Capacitor (qui réutilise le code React existant à ~80% plutôt qu'une réécriture complète en React Native). Permet d'utiliser l'application sur tablette en salle de consultation, avec distribution sur App Store et Google Play.
7
Interface multilingue
⏱️ Estimation : 3 à 4 semaines · Priorité : BASSE
Traduction de l'interface via la bibliothèque react-i18next (français → anglais dans un premier temps, puis d'autres langues selon la demande). Représente un chantier de fond car tous les textes sont actuellement écrits en dur dans le code.
📌 Prochaine action concrète recommandée : fournir l'anamnèse kinésiologique complète pour démarrer le plugin Kinésiologie et valider l'architecture de plugins avant de la généraliser à d'autres spécialités.

Architecture commerciale — Système de plugins

Concept : Core + Plugin

Le logiciel est divisé en deux parties distinctes : le Core (socle commun, identique pour toutes les spécialités) et le Plugin (anamnèse propre à chaque spécialité, vendu séparément ou bundlé dans l'installateur).

ComposantContenuModèle
Core (socle commun) Gestion patients · Calendrier RDV · Comptabilité · Facturation · Exports Excel/PDF · Sauvegardes chiffrées · Sécurité AES-256 · Mise à jour Inclus dans tous les packs
Plugin MTC Anamnèse MTC complète : pouls, langue, barrage, 5 éléments, points acupuncture, techniques, plantes, énergie… Pack MTC
Plugin Kinésiologie Anamnèse kinésiologique personnalisée (à définir) Pack Kinésiologie
Plugin Ostéopathie Anamnèse ostéopathique (à définir) Pack Ostéo
Plugin Naturopathie Anamnèse naturopathique, bilan nutritionnel (à définir) Pack Naturo

Mise à jour d'un plugin

Lorsqu'une anamnèse évolue (nouvelles questions, nouveau protocole), il suffit de distribuer le nouveau fichier plugin (.plugin) sans relivrer une nouvelle version du logiciel core. Le thérapeute l'importe depuis Paramètres → Mettre à jour mon formulaire.

🔵 Avantage clé : Le développement du core est fait une seule fois. Chaque nouvelle spécialité représente 3 à 5 jours de travail supplémentaire (création du plugin JSON) et non plusieurs semaines.
